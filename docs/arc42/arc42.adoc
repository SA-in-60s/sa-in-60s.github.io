== Einführung und Ziele

=== Aufgabenstellung

SA-in-60s ist eine Lernplattform für Software-Architektur-Konzepte. Jedes Konzept wird in einem 60-Sekunden-YouTube-Short erklärt. Die Website ergänzt die Videos um Lernpfade, Fortschrittstracking und KI-gestützte Vertiefung.

=== Qualitätsziele

[cols="1,2,1"]
|===
| Priorität | Qualitätsziel | Maß

| 1 | *Performance* — Seiten laden schnell, auch auf mobilen Geräten
| Lighthouse Performance >90

| 2 | *Accessibility* — Barrierefreier Zugang für alle Nutzer
| Lighthouse Accessibility 100

| 3 | *Offline-Fähigkeit* — Statische Seiten ohne Server-Abhängigkeit
| Keine Laufzeit-Abhängigkeit außer YouTube-Embeds

| 4 | *Datensparsamkeit* — Kein Tracking, keine Cookies, keine Server-Daten

== Querschnittliche Konzepte

=== Datenhaltung — localStorage

Alle Nutzerdaten liegen ausschließlich im Browser. Kein Backend, keine Cookies, keine Analytics.

[cols="1,1,2"]
|===
| Key | Typ | Inhalt

| `sa60s-lang` | String | "de" oder "en"
| `sa60s-progress` | JSON Array | Liste gelernter Konzept-IDs
| `sa60s-ai-chat` | String | "claude", "chatgpt" oder "perplexity"
| `sa60s-unlock` | JSON (base64) | Freischaltungszustand (Datum, Phase, Pfade)
|===

*Resilience:* Alle localStorage-Zugriffe sind in try-catch gewrappt. Bei fehlender Verfügbarkeit (Private Browsing) degradiert die Seite graceful — alle Inhalte sind zugänglich, nur Persistierung fehlt.

=== Internationalisierung (i18n)

Zwei Sprachen (DE/EN) über ein einfaches Attribut-System:

* HTML-Elemente tragen `data-de` und `data-en` Attribute
* `main.js` togglet Sichtbarkeit basierend auf aktiver Sprache
* Kein i18n-Framework nötig — Build-Zeit-Generierung
* Video-Embeds sind sprachspezifisch (separate YouTube-URLs)

=== Security

* *Content Security Policy* — Strikte CSP im HTML-Head: `default-src 'self'`, nur YouTube-Iframes erlaubt
* *Keine Nutzerdaten auf Server* — DSGVO by Design
* *HTML Escaping* — Alle dynamischen Inhalte via `escapeHtml()` in build.js
* *URL-Validierung* — YouTube-URLs strikt gegen Pattern geprüft
* *robots.txt* — Redirect-Seiten und Nicht-Stamm-Konzepte blockiert

=== Token-System für QR-Codes

Jedes Konzept hat einen deterministischen 5-Zeichen-Token (SHA-256 Hash aus ID + Salt). Tokens sind:

* Nicht erratbar (kein Muster erkennbar)
* Stabil (ändern sich nie, da deterministisch)
* Kurz genug zum Abtippen aus dem Buch
* Rückwärtskompatibel (ID-basierte URLs funktionieren weiterhin als Alias)

== Architekturentscheidungen

=== ADR-001: Statische Seiten statt SPA

*Status:* Akzeptiert

*Kontext:* Die Website muss auf GitHub Pages laufen (kein Server-Side-Rendering).

*Entscheidung:* Build-Zeit-Generierung aller HTML-Seiten via `build.js`. Vite bundled CSS/JS, build.js generiert 156+ HTML-Seiten.

*Konsequenzen:*

* (+) Maximale Performance, CDN-freundlich
* (+) SEO: Jede Seite hat eigene URL, Meta-Tags, JSON-LD
* (-) Kein dynamisches Routing — jede neue Seite muss gebaut werden
* (-) Content-Updates erfordern Rebuild + Deploy

=== ADR-002: localStorage statt Backend

*Status:* Akzeptiert

*Kontext:* Fortschritt, Sprache und KI-Präferenz müssen persistiert werden. Ein Backend wäre Overkill und DSGVO-relevant.

*Entscheidung:* Alle Nutzerdaten in localStorage.

*Konsequenzen:*

* (+) Kein Backend nötig, keine DSGVO-Probleme
* (+) Sofortige Persistierung ohne Netzwerk
* (-) Fortschritt geht beim Browserwechsel verloren
* (-) Private Browsing: kein Fortschritt
* (-) Kein geräteübergreifendes Sync

=== ADR-003: Kryptische Token-URLs für eBook-QR-Codes

*Status:* Akzeptiert

*Kontext:* QR-Codes im gedruckten eBook zeigen auf `/go/{id}` und `/ai/{id}`. Wenn IDs menschenlesbar sind (`/go/abstraktion`), können Leser das Muster erkennen und direkt URLs raten.

*Entscheidung:* 5-Zeichen SHA-256 Tokens (`/go/49d87`), menschenlesbare Aliases bleiben als Fallback.

*Konsequenzen:*

* (+) URLs nicht erratbar
* (+) Kurz genug zum Abtippen
* (+) Bestehende Links brechen nicht (Aliases)
* (-) Leicht höhere Komplexität in der Build-Pipeline

=== ADR-004: Unlock-System mit täglicher Freischaltung

*Status:* Akzeptiert

*Kontext:* 143 Konzepte auf einmal überfordern. Schrittweises Lernen ist effektiver.

*Entscheidung:* 1 Konzept pro Tag freischalten. Erst Stamm (10 Tage), dann gewählter Pfad.

*Konsequenzen:*

* (+) Nachhaltiges Lernen, keine Überforderung
* (+) Gamification-Effekt (täglicher Fortschritt)
* (-) Ungedulige Nutzer fühlen sich eingeschränkt
* (-) Ohne localStorage kein Gating (bewusste Entscheidung)

=== ADR-005: Lokale Mitternacht für Tageswechsel

*Status:* Akzeptiert

*Kontext:* ADR-004 definiert "1 Konzept pro Tag", aber nicht wann ein neuer Tag beginnt. UTC-basierte Tageswechsel führen dazu, dass Nutzer in verschiedenen Zeitzonen den Wechsel zu unerwarteten Uhrzeiten erleben — z.B. um 2:00 Uhr nachts in MESZ oder um 19:00 Uhr in EST. Das widerspricht der Erwartung "morgen gibt es ein neues Video".

*Entscheidung:* Der Tageswechsel erfolgt um lokale Mitternacht (00:00 Uhr) basierend auf der Zeitzone des Browsers (`new Date()` verwendet automatisch die lokale Zeitzone). Zusätzlich wird ein Countdown-Banner "Noch X Stunden bis zum nächsten Video" angezeigt, wenn alle Tages-Konzepte freigeschaltet sind.

*Implementierung:*

* Tagesdatum via `new Date().toLocaleDateString('sv')` (YYYY-MM-DD im lokalen Kalender)
* Vergleich mit gespeichertem Datum in `sa60s-unlock` (localStorage)
* Countdown berechnet Stunden bis nächste lokale Mitternacht: `new Date().setHours(24,0,0,0) - Date.now()`
* Bei Seitenbesuch nach Mitternacht: neues Konzept wird freigeschaltet

*Konsequenzen:*

* (+) Intuitive Erwartung: "morgen" = nach dem Schlafen
* (+) Kein Server nötig — Browser-API liefert lokale Zeit
* (+) Countdown gibt klare Erwartung wann es weitergeht
* (-) Zeitzonenwechsel (Reisen) kann unerwartete Effekte haben — akzeptabel
* (-) Nutzer können durch Verstellen der Systemzeit tricksen — bewusst ignoriert (localStorage-Manipulation wäre einfacher)

== Risiken und technische Schulden

[cols="1,2,2"]
|===
| Risiko | Auswirkung | Maßnahme

| *YouTube ändert Embed-API* | Video-Einbettung bricht | Fallback-Platzhalter, regelmäßig testen
| *localStorage voll* | Fortschritt geht verloren | Datenvolumen minimal (~1KB), Fehlermeldung
| *KI-Chat-URLs ändern sich* | Deep-Dive-Links brechen | Redirect-Seiten als Indirection, leicht anpassbar
| *eBook gedruckt mit falschen QR-Codes* | Links funktionieren nicht | Token-URLs sind deterministisch und stabil
| *Zu viele Konzepte für Graph* | Performance-Probleme | Cytoscape optimiert für 150+ Knoten, Pfad-Filter
|===
